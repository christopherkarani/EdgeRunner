# Lessons

- Bonsai Q1 work needs parity instrumentation before performance edits. Throughput-only experiments on this branch can look better while still producing garbage output.
- The raw `Q1_0_g128` layer path is currently the faster Bonsai path on this branch. Historical experiment notes from another code state were not reliable enough to reuse without re-benchmarking.
- Bonsai quality should be judged on realistic prompt generation, not only on pathological probes like `[1]` or forced-BOS continuations. Those probes are still useful for debugging, but they are not representative of normal use.
- Existing Q1 fused shaders are not automatically a throughput win. The current fused Q1 QKV, Gate+Up, and LM-head kernels regress Bonsai end-to-end decode badly on this machine, so every dormant kernel needs a full benchmark before it is promoted into the runtime path.
- Bonsai already has the same broad decode architecture ideas as llama.cpp: single-token decode, KV cache reuse, and fused decode kernels. When throughput is still far behind, the likely problem is kernel implementation quality or model-format-specific overhead, not the absence of those high-level patterns.
- Production entry points must not inject BOS outside the tokenizer contract. Bonsai disables automatic BOS in its GGUF metadata, so app-level forced-BOS logic in facade/session code silently breaks real generations even if low-level model tests look fine.
- When the user sets a throughput target, an architectural checkpoint is not a stopping point. Keep iterating through implementation and benchmark loops until the target is hit or a concrete blocker is proven with evidence.
- The Bonsai-only synchronous decode shortcut is not production-safe in the current codebase. Routing Bonsai greedy decode through `greedyTokenSync` caused hard crashes across quality, parity, and benchmark suites, so any future sync path needs dedicated validation before promotion.
- Bonsai LM head time is not the primary bottleneck on this machine. The measured LM head path is about `1.68 ms`, far below the roughly `27 ms/token` end-to-end decode cost at `~37 tok/s`, so the real throughput gap is in the Q1 per-layer decode stack.
- Selective Q1 kernel promotions need their own benchmark proof. A Bonsai-only fused Q1 QKV path was quality-safe but regressed throughput to about `25.9 tok/s`, so reducing dispatch count alone is not enough if the kernel shape sacrifices parallelism.
- The current shared Q1 GEMV kernel is already close to its local optimum for simple row-packing tweaks. Increasing its per-threadgroup row count from `2` to `4` preserved quality but still regressed Bonsai throughput, so the next Q1 speedup needs a different kernel design rather than a small constant tweak.
- When the user asks for a production-ready experimental path, do not collapse that into removal just because the current implementation is weak. Keep the safe baseline, but rebuild the experimental path as an explicit opt-in track with hard parity and benchmark gates.
- When the user explicitly wants the aggressive version of an experimental system, do not keep steering toward hybrid fallbacks. Use the baseline as a control, but focus the work on attribution and optimization inside the requested all-in design.
- If TurboQuant diverges at the post-attention residual of layer 0, do not waste time on FFN or late-layer fixes first. The next diagnostic target is the TurboQuant attention path itself: KV encode, score computation, and value reconstruction.
- A better rowwise value reconstruction metric does not guarantee a better model. If a TurboQuant V-path tweak lowers synthetic decode MSE but worsens pinned-model layerwise or smoke behavior, roll it back and target first-layer attention-output fidelity directly.
- If a proper dense-V control still leaves layer-0 attention far from the exact dense reference, stop tuning V in isolation. At that point the next all-in TurboQuant target is the K/score path, not another value-only heuristic.
- K-side synthetic improvements are not enough either. If a TurboQuant K/score tweak looks better on toy score/output MSE but worsens pinned layerwise or smoke behavior, roll it back immediately and treat the real model gates as the source of truth.
- When score-space instrumentation is available, prefer it over hidden-state proxies. If TurboQuant softmax weights are already saturating at layer 0 while `q8_0` softmax drift stays small, target the score contract directly before touching anything else.
- If a raw GPU-vs-CPU score-term probe matches to numerical tolerance, stop suspecting the Metal kernel first. At that point the failure is in the TurboQuant score contract itself or in how that contract interacts with real model activations.
- When `MTLCommandBufferErrorDomain Code=9` is opaque, rerun the failing path with `MTL_DEBUG_LAYER=1` immediately. The validation layer can turn “Invalid Resource” into the actionable message, like `buffer is not a MTLBuffer`, which is far more useful than the default error.
- If a randomized-Hadamard inner-product path is missing the inverse-dimension factor, fix that before any more heuristic tuning. A missing `1 / headDim` can create enormous score-space explosions that drown out every other experiment.
- Once TurboQuant score-formula error and decoded-key error collapse to the same value on the real model, stop tuning score constants. At that point the dominant problem is the key representation itself, so the next experiment must target key encoding/reconstruction fidelity.
- When CPU replay shows a different TurboQuant preset improves real-model key-score fidelity, promote that difference into the runtime as separate K/V layouts instead of continuing to force one shared contract. Mixed-layout support is a cleaner lever than more scalar tuning once representation fidelity is the bottleneck.
- After changing the TurboQuant contract, immediately retarget the low-level tests to the active contract. Passing metal tests on the old aggressive-only layout do not buy confidence for a `balanced64` / `balanced` rollout.
- Single-head TurboQuant attention tests are not enough for a GQA model. Add grouped `numHeads > numKVHeads` coverage before trusting low-level attention validation on Qwen-family models.
- If contract-aligned single-head and grouped-head TurboQuant tests both pass while the pinned model still diverges, stop treating “missing low-level coverage” as the main blocker. At that point the next diagnostic needs real captured model activations, not more synthetic signal tuning.
- If a fused prefill path writes `V` directly to a layer cache instead of the scratch buffer, trace collectors must read the real write target. Otherwise “exact activation” diagnostics can silently turn into all-zero false negatives.
- When introducing a more conservative packed layout, audit every Metal local buffer against the maximum code-word count before trusting the regression result. The old generic TurboQuant quantizer silently assumed `<= 16` words per row and corrupted higher-fidelity key layouts.
- When TurboQuant fresh-prefill and cached single-token decode disagree on the same token sequence, stop changing the format first. Add a direct cached-vs-fresh diagnostic and isolate the single-token decode kernel before drawing conclusions about the representation.
- In the TurboQuant single-token decode kernels, each lane owns a private 128-float accumulation slice. Zero the full slice, not just the lane-striped subset, before any softmax accumulation. Partial initialization there can masquerade as a cache/format bug.
- Once a configuration only works behind an env override, promote it into the default contract or it is not a production state. Verified residual-scale improvements should move from `EDGERUNNER_TURBOQUANT_KEY_RESIDUAL_SCALE` into `TurboQuantV2Contract.keyResidualScale`.
- If the pinned quality harness fails on a long repeated prefix, reconstruct that exact prefix state and compare TurboQuant fresh versus cached logits step by step. That quickly tells you whether the live bug is still in cached decode or has moved back into fresh-prefill fidelity.
- Do not assume long-prefix TurboQuant failures come from the generic multi-row quantizer just because short prompts pass. Add long-context low-level tests at the same scale as the failing prompt before changing kernels.
- If long-context quantization and standalone long-context attention both pass, stop re-litigating those generic kernels. The remaining blocker is either real-model activation fidelity or a model-path-specific prefill stage around those kernels.
- If higher-fidelity codebooks improve rowwise approximation, smoke, and layerwise metrics but leave the repeated-prefix quality harness unchanged, stop assuming the remaining problem is just bit budget. That points to a deeper limitation in the residual-sign codec family or how it interacts with real model activations.
- When claiming upstream or external precedent, distinguish clearly between audited source and unchecked assumption. In this project, `ggml-org/llama.cpp` and `TheTom/llama-cpp-turboquant` are different codebases; do not collapse them into one statement without checking the actual fork first.
- When the user says "work until it's rollout safe", only the pinned rollout gates count as progress. Repo plumbing, attribution tooling, or architecture cleanup do not justify returning if smoke, repeated-prefix quality, and long-context still fail.
- If the current branch is worse than a previously observed pinned-model result, stop exploratory optimization and do regression isolation immediately.
- After two unsuccessful tuning rounds without pinned-gate improvement, switch to a bisect-style hunt for the exact change that reintroduced failure.
- Repeated-prefix diagnostics can silently switch execution modes as the suffix grows. In this repo, the first extension step is true decode, but later steps become prefix-reuse suffix-prefill; do not describe the whole trace as "cached decode" without checking the control flow.
- After adding safety fallbacks, rerun the real rollout gates immediately. A stale failure report can survive in notes long after the branch has actually crossed the gate.
- Do not switch the TurboQuant transform basis from unnormalized randomized Hadamard to orthonormal fixed-sign WHT without auditing every dependent score/output scale at the same time. In this codebase, CPU/GPU agreement can still hold while the pinned model collapses because the end-to-end contract moved but some runtime assumptions did not.
- If TurboQuant smoke improves but the pinned 128-token quality harness still diverges early, do not treat the preset as "good enough." In this repo, a configuration can match the first greedy tokens and still collapse by step 2 unless the real-model key path and fork-style policy lifecycle are also correct.
- Fixed-type TurboQuant rows need fork-style corrected norm metadata too. The active Metal `turboquant_quantize_rows` path was still writing raw `rowNorm` and a non-zero residual norm for fixed `turbo2`/`turbo3`/`turbo4`, which silently diverged from both the local reference encoder and `llama-cpp-turboquant`.
- After changing the active TurboQuant contract, audit the test parameter builders as well as the kernels. Several attention tests were still hardcoding `turbo2` `codeWordsPerRow` / bit-width metadata, which created false `turbo3` regressions until the params were retargeted to the active preset.
- If pure-Turbo CPU replay and pure-Turbo GPU replay match on the same captured activations, stop blaming the Metal execution path. At that point a pinned-model failure is more likely a representation-fidelity problem in the active TurboQuant contract than a kernel desync.
- Fork-aligned InnerQ is opt-in, not an automatic `turbo3` default. If the local contract silently auto-enables InnerQ without `TURBO_INNERQ` or explicit local env, fix that first before using rollout failures to judge pure `turbo3`.
- If real-activation stored `turbo3` rows already match a fork-style reference at the code-word and corrected-norm level, stop spending time on row packing. At that point the remaining rollout gap is downstream model quality under the pure representation, not a hidden row-encode mismatch.
- If `KVCache` advertises per-layer storage kinds, audit the model runtime before trusting any mixed-policy result. In this repo the allocator already supported per-layer q8/Turbo splits, but prefill and decode still selected cache families as a single global mode, which made mixed-layer experiments invalid until the runtime loops were fixed.
- Early-layer q8 can improve TurboQuant quality without becoming rollout-safe. In this pinned path, forcing the first 2 layers to q8 lowered the 128-token max logit delta from `19.7134` to `16.8863`, but smoke still failed and TTFT stayed about `3.58x` worse than Q8, so partial recovery is not enough to call the policy production-usable.
- A true `q8 key + turbo value` hybrid is stronger than whole-layer q8 on this pinned model. Forcing the first 2 layers to hybrid `K=q8 / V=turbo` kept the first 5 decode steps aligned with Q8 and lowered `max_abs_logit_delta` further to `11.8762`, so the dominant live error really is key fidelity more than value fidelity.
- Even when key-only q8 materially improves quality, do not confuse that with rollout safety. On this repo the best `K=q8 / V=turbo` candidate still failed smoke and still had TTFT around `107.6s` vs `32.3s` for Q8 at 4k, so hybrid improvement is evidence about the blocker shape, not proof of a production fallback.
- If a hybrid `K=q8 / V=turbo` override moves the first divergent attention output from layer 0 to layer 2, treat that as evidence that the layer-0 key blocker is real but not sufficient. In this pinned rollout the best 2-layer hybrid matched Q8 for the first 5 decode steps and still failed later, which means the remaining blocker lives deeper in the still-Turbo layers rather than in the original layer-0 collapse alone.
- The pure-Turbo replay harness cannot be reused blindly under hybrid q8-key experiments. In this repo it throws `unsupportedBitWidth(0)` as soon as a replayed layer stack contains q8-key layers, so hybrid isolation must use the live layerwise input/trace path or be made layer-storage-aware first.
- If promoting the first 3 key layers to q8 still leaves `first_divergent_layer=2` while the first attention residual divergence moves to layer 3, stop extending the fallback depth. That pattern means the blocker is now compounded hidden-state drift across the remaining pure-Turbo stack, not a single next layer waiting to be promoted.
- After editing Swift tests, do not trust `swift test --skip-build`. It will happily run the stale binary and can make an invalid diagnostic look real until the targeted test is rebuilt.
