# Thread — When Your Optimization Swarm Finds Nothing — 2026-03-24

## 1/

I built a 6-agent autonomous research swarm to optimize my Metal LLM inference engine.

The result after 3 days? 228 → 230.8 tok/s.

Not a typo. Two failed experiments, one 1.2% gain, and a hard lesson about optimization work.

## 2/

The setup:
- EdgeRunner: Swift/Metal inference for Qwen 0.6B
- Baseline: 228 tok/s decode on M3 Max
- Target: Beat MLX at 277 tok/s
- Method: Multi-agent swarm with scientific rigor

## 3/

The swarm architecture:

🔬 Researcher — Whitepapers, llama.cpp/MLX source
🎯 Designer — Falsifiable hypotheses with quantified predictions
🔧 Implementer — Surgical code changes
📊 Benchmarker — Statistical rigor (10+ runs, p-values)
🤔 Analyst — Keep/rollback decisions
📝 Logger — Complete experiment database

Each agent with one job. No context pollution.

## 4/

Experiment 33: SIMD-optimized softmax

Hypothesis: Replace threadgroup barriers with simd_sum for 10-15% speedup
Mechanism: Faster parallel reduction in GQA attention
Predicted: 228 → 250 tok/s

Result: Wrong tokens. Correctness failed.

## 5/

The SIMD softmax bug was subtle. Online softmax requires careful renormalization when combining per-thread accumulators across a simdgroup.

My "optimization" broke the numerical stability. The tokens went from [1, 1479, 35, 5371, 1] to [1, 101828, 122053].

Rolled back.

## 6/

Experiment 34: Metal Indirect Command Buffers

Hypothesis: Pre-record 142 dispatches, eliminate CPU encoding overhead
Mechanism: Record once, replay with buffer offset updates
Predicted: 228 → 260 tok/s (+14%)

## 7/

The blocker:

```swift
// iOS/tvOS:
icbDesc.commandTypes = .concurrentCompute  // ✅

// macOS:
// ❌ MTLIndirectCommandType has no member 'compute'
```

ICBs are render-focused. Compute commands? iOS only.

Lesson: Read the SDK headers *before* you design the experiment.

## 8/

Experiment 35: Tile-based GEMV

Hypothesis: Cooperative tile loading → 20% bandwidth improvement
Mechanism: 32 threads load contiguous tile into threadgroup memory
Predicted: 228 → 275 tok/s

## 9/

Actual result: 230.8 tok/s (+1.2%)

The existing Q8_0 GEMV kernel already had reasonable memory access patterns. Threadgroup synchronization overhead ate most of the gains.

Kept the code — it's cleaner — but no breakthrough.

## 10/

The hard truth from profiling:

- GPU time: 3.4ms per decode (fixed — bandwidth limited)
- Swift async overhead: 0.23ms
- Inter-dispatch idle: ~0.5ms
- TLB/page overhead: the rest

The bottleneck isn't kernel efficiency. It's architecture.

## 11/

What would actually get to 280 tok/s?

1. Protocol changes to eliminate async/await per token
2. Metal ICBs (but macOS doesn't support compute ICBs)
3. Fundamental memory layout changes

Not micro-optimizations. Structural work.

## 12/

So the autoresearch swarm "failed" to hit the target.

But it succeeded at exactly what it was designed to do: falsify hypotheses quickly, document everything, and tell me where the real ceiling is.

## 13/

The scientific method doesn't guarantee breakthroughs.

It guarantees you'll know when you don't have one. That you won't ship broken optimizations. That you'll understand *why* the ceiling is where it is.

That's worth more than fake metrics.
